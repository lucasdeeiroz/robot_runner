# ADB Communication & Telemetry Best Practices

Você está implementando comunicação com dispositivos Android via Android Debug Bridge (ADB), rotinas de telemetria ou ferramentas de inspeção do Robot Runner AI.

1. **ADB Command Grouping (Batching)**: A latência do USB somada ao tempo de spawn de múltiplos processos locais do `adb.exe` prejudica severamente a performance. Sempre que precisar recuperar múltiplas métricas simultâneas (bateria, memória, tela, CPU), crie uma única super-string e despache no terminal (ex: `adb shell "cmd1 ; cmd2 ; cmd3"`).
2. **Separação Blindada de Output (Escaping)**: Ao fazer batching de comandos no shell, precisamos separar os retornos para o parsing posterior no backend. Use a instrução `echo` combinada a separadores stringentes com escape em aspas (`___S\"EP\"___`). Ambientes como o Windows (PTY/CMD) frequentemente causam *shell echo* de volta (ecoam a linha de comando); o escape previne que o código Rust faça split prematuro na sujeira do terminal, já que o retorno real terá o delimitador resolvido (`___SEP___`).
3. **Prevenção do "Observer Effect"**: Monitorar o dispositivo constantemente afeta seu estado basal, consumindo CPU, inflando RAM com dumps de string massivos e descarregando a bateria (Heisenbug). Limite o output das ferramentas ADB à menor superfície de texto necessária:
   - *Nunca* use flags verbosas de depuração (ex: `dumpsys gfxinfo <pkg> framestats`) se não for estritamente necessário. O comando comum `dumpsys gfxinfo <pkg>` já fornece estatísticas reduzidas suficientes (como total de frames) sem inundar o buffer de IPC (Inter-Process Communication).
4. **Precisão Cumulativa vs Instantânea (CPU)**: Evite surpresas com o utilitário `top`. O Android nativamente reporta a CPU cumulativa desde o boot no primeiro ciclo do `top`. Se você precisa de estatísticas CPU *instantâneas* (real-time) precisas de um processo, você é forçado a usar delays com duas iterações (ex: `top -b -n 2 -d 0.5`), o que inevitavelmente criará bloqueios de ~0.5s de overhead e consumirá recursos de bateria/CPU do alvo.
5. **Filtro de Ruído do Ciclo de Vida ADB**: Nunca confie na primeira ou última linha gerada pelo `adb.exe`. Sempre varra a string do `stdout` via código para remover lixos gerados pelo daemon de inicialização (ex: descartar todas as linhas que comecem com `* daemon` ou `adb server`). Se não o fizer, retornos vazios ou dados corrompidos poluirão a lógica de negócios da ponte Rust/React.
6. **Descoberta de App em Primeiro Plano (Foreground Activity)**: O comando `dumpsys activity top` é extremamente oneroso e pode demorar mais de 10 segundos em dispositivos ocupados (despejando o estado hierárquico inteiro da Activity). Para apenas descobrir qual é a atividade/pacote em foco, prefira sempre o `dumpsys window displays`, que leva menos de `200ms` e expõe a mesma informação nas propriedades `mCurrentFocus` ou `mFocusedApp`.
